# 用户端首页视差滚动设计文档

## 背景

当前用户端首页已经具备较完整的视觉结构：首屏 Hero、文章网格、右侧边栏、页脚品牌区，以及 `HeroCosmicRing` 装饰组件。现状问题不在于页面内容不足，而在于整页滚动时层次感主要依赖静态布局与局部入场动画，缺少贯穿首页的连续滚动深度。

本次需求是为用户端首页增加“克制、顺滑、接近原生质感”的整页视差滚动。效果必须覆盖整页，而不是只停留在首屏 Hero；同时移动端也必须保留效果，但幅度要更轻，优先保证阅读稳定性与滚动性能。

## 目标

- 为首页建立整页一致的视差滚动体验，提升滚动层次感。
- 让 Hero、文章区、侧边栏和页脚在滚动中保持轻量级前后景关系。
- 保持黑夜模式可用、移动端可用、性能稳定、可访问性可退化。
- 尽量不改动首页现有数据获取逻辑、路由逻辑和真实业务流程。

## 非目标

- 不重做首页信息架构，不新增业务模块。
- 不引入外部动效库。
- 不做夸张的 3D 翻转、吸附滚屏、整页缩放或明显眩晕式位移。
- 不改成假数据驱动的视觉演示首页。

## 当前实现观察

- 首页入口位于 [index.vue](D:/code/sidifensen_blog/sidifensen_blog_frontend/sidifensen_user/src/views/Home/index.vue)。
- Hero 区域已存在独立视觉装饰组件 [HeroCosmicRing.vue](D:/code/sidifensen_blog/sidifensen_blog_frontend/sidifensen_user/src/components/hero/HeroCosmicRing.vue)。
- 页面已经有两类展示动画：
  - 首屏副标题打字机动画
  - 文章卡片基于 `IntersectionObserver` 的入场动画
- 页面当前使用局部 CSS 变量控制明暗主题，且首页样式集中在同一单文件组件中。

这些观察意味着：新增视差逻辑应优先做“页面级统一驱动”，减少把滚动监听散落到各块中的风险。

## 方案选择

最终选型为“分层慢速视差”。

备选方案对比：
- 分层慢速视差：整页多个层以不同小幅速度响应滚动，观感最克制，最接近原生滚动质感。
- 区块吸附式视差：更强调区块节奏，但设计感偏强，不符合“克制型”要求。
- 卡片流体微悬浮：风险最低，但整页视差存在感不足。

选择理由：
- 能覆盖整页而不止首屏。
- 便于在桌面端和移动端按同一模型做不同幅度调优。
- 更容易与现有入场动画和黑夜模式共存。

## 视觉与交互设计

### 1. 分层模型

首页划分为四个视差层组：

1. 背景层
   - 目标元素：`hero-bg`、页脚顶部装饰线、必要时主内容浅背景修饰
   - 特征：移动最慢，负责建立页面深度基线

2. 装饰层
   - 目标元素：`HeroCosmicRing` 及其内部星环/信息卡/卫星标签整体容器
   - 特征：比背景层略快，形成首屏空间感

3. 内容层
   - 目标元素：Hero 标题、副标题、按钮、导航胶囊、主内容区标题
   - 特征：位移最轻，保证文字阅读稳定

4. 内容块层
   - 目标元素：文章网格、右侧边栏、页脚品牌与导航组
   - 特征：不同模块采用轻微差速，滚动中仍保持整页层次

### 2. 动效原则

- 所有位移均以纵向平移为主，不加入明显旋转、缩放或强透视。
- 位移是“持续响应滚动”的轻微层移，而不是进入视口时突然触发的大幅动画。
- 内容层的速度差必须小于装饰层，避免文字漂浮感过强。
- 文章卡片原有入场动画继续保留，但不能与视差位移叠加得过于明显。

### 3. 移动端策略

- 移动端保留视差，不直接关闭。
- 缩小位移幅度。
- 减少同时运动的层数或复杂度。
- 保证页面滚动与点击命中区域稳定，不因为 transform 造成明显误触或布局跳动。

### 4. 可访问性策略

- 当系统开启 `prefers-reduced-motion: reduce` 时，视差应禁用或降到近乎静止。
- 打字机动画和已有装饰动画不在本次范围内强制重构，但新加的视差必须遵守该策略。

## 技术设计

### 1. 统一滚动状态源

首页建立一个统一的滚动状态源，而不是让 Hero、主内容区、页脚各自监听滚动事件。

推荐做法：
- 在首页组件中注册单一 scroll 监听。
- 通过 `requestAnimationFrame` 合并滚动更新，避免高频同步写样式。
- 将计算结果写入页面级 CSS 变量或样式对象，再由各视差层读取。

这样做的好处：
- 更新时机集中，方便控帧。
- 便于统一调整不同层的速率和最大位移。
- 后续如果要拆分 composable 或子组件，迁移成本更低。

### 2. CSS 变量驱动

推荐为每类层定义专属 CSS 变量，例如：
- `--parallax-hero-bg-y`
- `--parallax-hero-ornament-y`
- `--parallax-hero-content-y`
- `--parallax-main-y`
- `--parallax-sidebar-y`
- `--parallax-footer-y`

各层使用 `transform: translate3d(0, var(...), 0)` 读取，不直接在多个地方重复计算像素值。

约束：
- 只对需要视觉位移的容器应用 transform。
- 不让 transform 作用在会影响布局测量和点击区域的最外层结构上。

### 3. 与现有动画的关系

- `IntersectionObserver` 控制的文章卡片入场动画继续保留。
- 视差作用应优先落在文章网格容器或列级容器，而非每张卡片都单独追加滚动位移。
- `HeroCosmicRing` 自带的内部循环动画保留，本次新增的是其整体相对页面的滚动层移。

### 4. 性能边界

- 所有滚动计算需避免读取过多布局信息。
- 优先基于 `window.scrollY`、视口高度和静态倍率完成计算。
- 避免在滚动回调中频繁 `querySelectorAll`。
- 避免对大量文章卡片逐个绑定视差样式。

## 测试与验证设计

### 自动验证

- 运行用户端检查命令，确保没有语法或类型检查错误：
  - `cd sidifensen_blog_frontend/sidifensen_user && npx vite-plugin-checker check`
- 运行用户端构建：
  - `cd sidifensen_blog_frontend/sidifensen_user && npm run build`

### 手动验证

- 桌面端首页滚动时，Hero 背景、装饰层、内容层和主内容区存在轻微差速。
- 文章区和右侧边栏继续具有层次感，但不影响点击与阅读。
- 黑夜模式下视觉层次仍成立，没有因为叠加透明度导致脏灰或发虚。
- 移动端滚动仍保留视差，但不产生明显卡顿和晕动。
- 开启系统减少动态效果时，新增视差不再明显运动。

## 风险与缓解

- 风险：首页文件已较大，继续堆叠逻辑会降低可维护性
  - 缓解：如滚动计算超过轻量级范围，抽离 composable

- 风险：文章卡片原有入场动画与整页视差叠加导致视觉过重
  - 缓解：把视差重心放在网格容器，而不是卡片逐个偏移

- 风险：移动端 transform 过强导致滚动发飘
  - 缓解：移动端统一降幅，并减少层数

- 风险：`prefers-reduced-motion` 未兼容导致可访问性问题
  - 缓解：在新增视差层统一提供降级样式

## 实施完成判定

满足以下条件即可认为设计完成：
- 首页整页存在连续、克制的视差层次感
- 桌面端与移动端均保留效果
- 黑夜模式与现有真实数据请求不受影响
- 性能验证与构建验证通过
- 视差逻辑边界清晰，没有把首页进一步变成难维护的杂糅脚本
